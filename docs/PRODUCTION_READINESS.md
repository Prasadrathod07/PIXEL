# Production Readiness — Pixel Issue Tracker

This document covers the security, reliability, scalability, and deployment considerations for a production deployment. Not every item below is implemented in the current build - the goal is to demonstrate clear awareness of what a production system requires and how each gap would be closed.

## Security

### Authentication and Sessions

Supabase Auth issues JWTs stored in HTTP-only cookies. These are not accessible to JavaScript and are not vulnerable to XSS-based token theft. Token refresh is handled automatically by the Supabase client library.

Recommended addition: enable `refresh_token_rotation_enabled: true` in Supabase Auth settings to invalidate refresh tokens after single use.

### Authorization — Two Independent Layers

**Application layer (proxy.ts)**
Route-level protection. Unauthenticated requests redirect to `/login`. Manager routes reject non-manager sessions. Client routes reject unauthenticated access.

**Database layer (Row Level Security)**
Data-level protection. Every table has RLS policies enforced at the query level, independent of the application layer. A client cannot read another client's issues even if application-level auth is bypassed. RLS is the final guard — it cannot be circumvented by application bugs.

This double enforcement means a compromise of one layer cannot expose data.

### Secret Management

- All keys stored in `.env` files, excluded from version control via `.gitignore`
- `SUPABASE_SERVICE_ROLE_KEY` is never prefixed with `NEXT_PUBLIC_` and never sent to the browser
- Service role key used only in server-side API routes for cross-user operations
- Production: store all secrets in Vercel environment variables (frontend) and Railway environment variables (AI service)
- Rotate API keys on a defined schedule

### Input Validation

- API routes validate required fields and reject malformed requests before touching the database
- Supabase uses parameterized queries throughout — SQL injection is not possible by design
- Recommended addition: Zod schema validation on all API route request bodies for type-safe runtime validation

## Reliability

### Graceful AI Degradation

The AI service is deliberately decoupled from the core application. If the AI service is unreachable, the Next.js API proxy returns 503 and the frontend displays "AI analysis unavailable" without crashing. Issues can still be created, managed, and resolved without AI features. AI is an enhancement, not a dependency.

### Error Handling

- All API routes return consistent JSON: `{ "error": "message" }` with correct HTTP status codes
- 401 unauthenticated, 403 unauthorized, 404 not found, 500 server error
- The AI service returns structured fallback responses if the LLM call fails, with `confidence_score: 0.5` as a signal to the frontend that the response is generic rather than model-generated

### Logging

Current: `console.log` / `console.error`

Production:
- Winston for structured JSON logging in the Next.js layer
- Python `logging` module with JSON formatter in the AI service
- Datadog or Logtail for log aggregation and search
- Sentry for error tracking with stack traces, user context, and release tracking

### Health Checks

- AI service: `GET /health` endpoint returns service status, model name, and embedding model
- Recommended: `GET /api/health` in Next.js that verifies Supabase connectivity and returns response time

### Backup Strategy

**Database (Supabase):**
Supabase free tier runs daily backups with a 7-day retention window. Supabase Pro extends this to point-in-time recovery (PITR) with up to 30 days of retention. For this application, the critical data is `issues` and `timeline_events` — losing either means losing client communication history, which is unacceptable in production.

Recommended for production:
- Enable PITR on Supabase Pro
- Run weekly `pg_dump` exports to S3 or GCS as a secondary backup independent of Supabase's infrastructure
- Test restore quarterly — a backup that has never been restored is not a backup

**pgvector embeddings:**
Embeddings are derived data — they can be regenerated by re-running the AI pipeline over existing issue text. They are included in the standard database backup. If embeddings are lost, regeneration requires one API call per issue, which is straightforward to script but takes time at scale.

**AI service (Railway):**
The FastAPI service is stateless — no data lives on the container. Railway redeploys from the GitHub repo automatically. Recovery from a container failure is a git push away.

**Environment variables / secrets:**
Secrets are not in version control. In a team context, they must be stored in a secret manager (Doppler, 1Password Secrets Automation, or AWS Secrets Manager) and not only in the hosting provider's UI. A hosting provider account compromise should not be the only place a secret exists.

In-app notifications are delivered live via Supabase Realtime. Email notifications are console-logged as a mock:

### Production Implementation

```bash
npm install resend
```

```typescript
import { Resend } from 'resend'
const resend = new Resend(process.env.RESEND_API_KEY)

export async function sendIssueResolvedEmail(params: {
  to: string
  clientName: string
  issueTitle: string
  issueUrl: string
}) {
  return resend.emails.send({
    from: 'notifications@pixelagency.com',
    to: params.to,
    subject: `Issue Resolved — ${params.issueTitle}`,
    html: resolvedEmailTemplate(params),
  })
}
```

The notification utility at `lib/utils/notifications.ts` is already structured to accept this swap. Replace the `console.log` block with the Resend call and add `RESEND_API_KEY` to environment variables.

Resend free tier: 3,000 emails/month. Sufficient for a small agency.

## Scalability

### Database

- pgvector ivfflat index with `lists = 100` performs well to approximately 1 million vectors. Beyond that, increase `lists` proportionally or switch to the HNSW index type for better recall at scale.
- Supabase Pro adds PgBouncer connection pooling — essential when deploying multiple Next.js instances against the same database.
- Read replicas available on Supabase Pro for separating read and write workloads.
- All high-frequency query columns are indexed: `status`, `severity`, `created_at`, `client_id`, `site_id`.

### Realtime Notifications

- Supabase Realtime: 500 concurrent connections on free tier, 10,000 on Pro
- Each connected browser holds one WebSocket channel per user
- For significantly higher scale: migrate to a dedicated WebSocket server with database webhooks as the event trigger

### AI Service

- FastAPI is stateless and horizontally scalable behind a load balancer
- BGE-M3 embeddings are deterministic for identical input — cacheable by content hash to reduce HuggingFace API calls
- LiteLLM supports load balancing across multiple API keys and providers natively, with automatic failover

## Deployment

### Frontend — Vercel

1. Push repository to GitHub
2. Import project in Vercel dashboard
3. Add all environment variables
4. Deploy — automatic on every push to `main`

### AI Service — Railway

```dockerfile
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

1. Create new Railway project from GitHub repo
2. Point to `ai-service/` directory
3. Set environment variables
4. Railway detects Dockerfile automatically
5. Copy the public URL to `NEXT_PUBLIC_AI_SERVICE_URL` in Vercel

### CI/CD — GitHub Actions

```yaml
name: CI
on: [push]
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '18' }
      - run: cd web && npm ci
      - run: cd web && npx tsc --noEmit
      - run: cd web && npm audit --audit-level=high
      - run: cd web && npm run build
```
